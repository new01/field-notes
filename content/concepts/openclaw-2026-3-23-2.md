OpenClaw 2026.3.23-2 is a patch release following 2026.3.23. Where the previous release targeted browser attach reliability and ClawHub auth, this one is dominated by security hardening (sandbox escapes, SSRF, spawn boundaries) and operational fixes for multi-account channel routing, Feishu reliability, Telegram edge cases, and cron scheduling. If you run OpenClaw in production with multiple channel accounts or cron jobs, this update is worth prioritizing.

## Security: sandbox escape and SSRF fixes

The most critical changes in this release are security fixes reported by @tdjackey and others. Several of these close real escape paths that could allow an attacker to write outside the sandbox or reach internal services.

**Sandbox media TOCTOU escapes** — Two separate symlink-retarget time-of-check-time-of-use vulnerabilities in sandbox media handling are fixed. The first affected attachment and image loads: a symlink could be retargeted between the boundary check and the actual read, allowing reads outside the sandbox root. The second affected `stageSandboxMedia` — a symlink at the destination path could redirect copies outside the workspace via the `media/inbound` alias. Both are fixed by using root-scoped boundary-safe reads and writes that verify identity after opening.

**Workspace safe writes** — `writeFileWithinRoot` was vulnerable to a symlink-retarget TOCTOU race between file open and truncation. The fix opens existing files without truncation, creates missing files with exclusive create, and defers truncation until post-open boundary validation passes. Out-of-root create artifacts are cleaned up on blocked races.

**SSRF hardening** — Web search citation redirect resolution (Gemini) now enforces strict SSRF defaults, blocking redirects to localhost, private, and internal targets. Separately, DNS pinning for `web_fetch` and citation redirects is preserved even when proxy env vars are set — previously, proxy configuration could bypass SSRF guards for untrusted URLs.

**Subagent sandbox inheritance** — Two fixes prevent sandbox boundary downgrades through subagent spawning. Sandboxed sessions can no longer spawn cross-agent subagents that would run unsandboxed, and ACP spawns from sandboxed requesters are rejected (as are `sandbox="require"` ACP runtime requests).

**Node metadata policy** — Node platform classification is hardened against Unicode confusable attacks. Unknown platforms now default to a conservative command allowlist that excludes `system.run` and `system.which` unless explicitly permitted.

**Browser control auth** — If browser-control auto-auth setup fails and no explicit token or password exists, the control server now aborts instead of starting unauthenticated.

These are the kind of fixes where the changelog entry matters more than the feature description. If you expose OpenClaw to untrusted inputs (which most production setups do), update promptly.

## Multi-account channel routing: defaultAccount support

A long-standing operational friction point for multi-account setups: when a message tool call omitted `accountId`, OpenClaw routed to whichever account happened to be first in config ordering. This was unpredictable and led to messages going out from the wrong account.

This release adds `channels.<channel>.defaultAccount` across message channels. When set, omitted `accountId` routes to the configured default instead of relying on implicit ordering. Fallback behavior is unchanged when unset, so existing configs are unaffected.

Feishu gets the deepest treatment here — `channels.feishu.defaultAccount` is wired into outbound routing with schema validation, and provider-prefixed targets (`feishu:` / `lark:`) are normalized correctly. Telegram multi-account gets a related fix: non-default accounts now fail closed when route resolution falls back to `matchedBy=default`, preventing cross-account DM and session contamination.

## Feishu reliability: replies, dedupe, and topic routing

Feishu received a cluster of fixes that together address most of the reliability complaints from multi-bot and topic-based Feishu deployments:

- **Quoted-message extraction** keeps text-first formatting (post/interactive/file placeholders instead of raw JSON), and inbound preview text no longer leaks into prompt system events
- **Video sends** now route as `msg_type: "file"` instead of failing silently
- **WebSocket event handling** is non-blocking in monitor dispatch, eliminating a class of event processing stalls
- **Topic session routing** uses `thread_id` as a fallback when `root_id` is absent, keeping first-turn topic keys stable across thread creation
- **Reply fallback** — when reply targets are withdrawn or deleted, sends fall back to normal delivery instead of failing
- **Dedup restart resilience** — persistent dedup state is warmed into memory on monitor startup, so retry events after gateway restart stay suppressed
- **Doc create permissions** — caller-controlled owner fields are removed from `feishu_doc` create; optional grant behavior is bound to trusted requester context only

## Telegram: threading, proxy, and media edge cases

Telegram fixes in this release are mostly edge-case corrections that collectively improve reliability for complex setups:

- **Proxy support restored** — The Node 22 `autoSelectFamily` workaround was breaking `HTTP_PROXY`/`HTTPS_PROXY` for outbound Telegram requests. Fixed by using `EnvHttpProxyAgent` instead of replacing the global dispatcher
- **Media fetch IPv4 fallback** — Telegram media fetches retry with IPv4-first settings when dual-stack connect errors occur, improving reliability on broken IPv6 routes
- **DM topic session isolation** — DM topic thread sessions are now scoped by chat ID (`<chatId>:<threadId>`) so parallel DMs cannot collide on shared topic IDs
- **Thread reply fallback safety** — `message thread not found` retries without `message_thread_id` only for DM threads, not forum topics
- **Voice caption overflow** — When `sendVoice` captions exceed length limits, the voice is re-sent without caption and text is delivered separately
- **Multi-account group isolation** — Channel-level `groups` config no longer leaks across accounts in multi-account setups
- **Polling teardown on restart** — Old long-poll loops are fully torn down on SIGUSR1 restart, reducing post-restart `getUpdates` 409 conflict storms

## Cron scheduling: failure alerts, timer fixes, and isolation improvements

Cron gets meaningful operational improvements:

**Failure alerting** — Configurable repeated-failure alerts with per-job overrides, including `failureAlert.mode` (announce or webhook), account routing, and Web UI editor support. You can now set thresholds and cooldowns for alert delivery instead of getting silent failures or noisy per-failure notifications.

**Timer hot-loop guard** — Stale past-due jobs could trigger repeated `setTimeout(0)` loops, saturating the event loop and flooding logs. A minimum timer re-arm delay now prevents this.

**Isolated session fixes** — Isolated cron runs now correctly resolve `subagents.model` through allowlist-aware selection, persist the intended model/provider on session entries for `sessions_list`, and avoid reusing stale CLI session IDs that caused premature timeouts at roughly one-third of configured `timeoutSeconds`.

**Session target guardrail** — Creating or patching `sessionTarget: "main"` cron jobs for non-default agents is now rejected at write time, preventing invalid cross-agent main-session bindings.

## Discord and Slack: reconnect integrity and auth fixes

**Discord reconnect watchdog** — A shared armable transport stall-watchdog detects silent close/reconnect zombies in the Discord gateway lifecycle. This addresses a class of issues where Discord connections would silently die and never recover.

**Slack socket auth failure handling** — Non-recoverable auth errors (`account_inactive`, `invalid_auth`) during startup and reconnect now fail fast instead of retry-looping indefinitely.

**Slack security ingress guard** — Slash-command and interaction payloads are now dropped when app/team identifiers don't match the active Slack account context, preventing cross-app or cross-workspace payload injection.

**Slack bot attachment messages** — Bot messages with empty `text` now include attachment `text`/`fallback` content when `allowBots: true`, so webhook alerts are no longer silently dropped.

## Gateway and CLI hardening

- **Config validation CLI** — `openclaw config validate` (with `--json`) validates config files before gateway startup, with detailed invalid-key paths in errors
- **CLI startup speed** — Plugin preload is skipped on fast routes, `--version` gets a fast-path bypass, and Node compile cache is enabled at startup. Meaningful improvement on Raspberry Pi and small hosts
- **Docker compose** — `openclaw-cli` runs in the gateway service network namespace with `cap_drop` and `no-new-privileges`, and the shared trust boundary is documented explicitly
- **macOS/iCloud state warning** — Doctor now warns when OpenClaw state resolves inside iCloud Drive, which can cause lock/sync races
- **Gateway websocket security** — Plaintext `ws://` is loopback-only by default, with explicit opt-in required for private-network access

---

2026.3.23-2 is a security-and-reliability patch. The sandbox escape and SSRF fixes alone justify updating promptly for any production deployment. The multi-account routing, Feishu reliability, and cron improvements are operational quality-of-life that reduce the kinds of issues that only surface after days of continuous multi-channel operation.
